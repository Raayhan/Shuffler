1. Introduction

1.1 Purpose

The software design specification is made with the purpose of outlining the software architecture and design of the "Shuffler" app in detail. The document will provide developers an insight in meeting clients' needs efficiently and effectively. Morever the document facilitates communication and understanding of the software by providing several views of the system design.

1.2 Scope

The software design document is to demonstrate how the design will accomplish the functional and non-functional requirements captured in the Software Requirement Specification (SRS). The document will provide a framework to the programmers through describing the high level components and architecture, sub systems, interfaces, database design and algorithm design.This is achieved through the use of architectural patterns, design patterns, class diagrams, database design etc.

1.3 Intended Audience

This document is mainly for the developers so that they can get a detailed idea about how the software need to be developed.

1.4 Definitions, Acronyms, and Abbreviations

2. Architectural Design

2.1 High Level Components and Their Interactions

User Component:

This is one of the key components of "Shuffler". This is where the user's actions, area activities will be implemented. This is also includes user account information.

Authentication and user management component:

This is the major sub system that is responsible for the security of the app. It authenticates users and also handles the user management activities such as creating new user accounts, removing acounts and etc.

History Component:

Here the user's recent history will be implemented. One user will have one history. It will be synchronized with server when a user logged in with a new device.

Activity Component:

In the activity component, all activities of a specified area will be saved into the database. User will be able to select a specific location and see the activities within an area around that location.

2.2 Architectural Styles/Patterns

The app will be developed under MVC (Model-View-Controller) architectural style/pattern. Users will be able to access the application with active internet connection within any location in the world.

MVC Architectural Style (Model-View-Controller)

MVC style separates presentation and interaction from the system data. The system is structured into three logical components that interact with each other:
  1. The Model component - Manages the system data and associated operations on that data.
  2. The View component - Defines and manages how the data is presented to the user.
  3. The Controller component - Manages user interaction and passes these interactions to the View and the Model.

2.3 Physical Arrangements of Device and Communication

The app deals mainly with hardware devices and installed software components on devices. The system performs many tasks. It consists of navigation based system used by users. The system helps to record user's name, date of birth, email, phone number, address etc.
Internet is the worldwide interconnection of all smart communication devices that have a valid IP. An active internet connection in mandatory to use the application.

2.4 Design Decisions Apllied in Whole Application

Object Oriented Software Development Methods

Reasons:
  1. Improved software maintainability.
  2. Faster development.
  3. Lower cost development.
  4. Improved software development productivity.
  5. Higher quality software.

MVC Architectural Pattern

Reasons:
  1. It should interact with other machines or users effectively.
  2. For more efficient interaction, the system should have flexible interfaces.
  3. MVC can be taken as a popular and easy to handle web application development style that has the feature of separating the presentation, business & intermediate logics.
  4. Ease to coding and provide well defined interfaces within each logic.

6. Database Design

6.1 Design Objective

A well-designed database should:

  • Support both required and ad hoc information retrieval.
  • Eliminate data redundancy, duplication and inconsistencies.
  • Ensure data integrity and accuracy.
  • Support business rules relevant to the organization.
  • Lend itself to future growth and development.

6.2 Table Organization

6.3 Relationships

  • Users and History: Each user can select multiple activities to be saved in their history, and therefore the UserID, which is the primary key for the Users table, is one half of the composite foreign key for the History table, making the Users table have a One-to-Many relationship with the History table.
  • Activities and History: Each activity can be selected by multiple users to be stored in their history, and therefore the ActivityID, which is the primary key for the Activities table, makes the other half of the composite foreign key for the History table, making the Activities tables have a One-to-Many relationship with the History table as well.

6.4 Normalization

All three tables in our database, Users, History and Activities each have all their non-key columns fully dependent on their primary key only, and the non-key columns are also independent of each other. Hence, all the tables are in Third Normal Form (3NF).

6.5 Scope of Expansion

There are further plans of expanding an activity to also have pictures and icons, which will be transformed to another table with the foreign key of the Activities table, and is therefore perfectly safe.

7. API Listing & Details

The app will primarily be dependent on 2 APIs:

  1. Google Maps JabaScript API: This API will allow our app to have a Google Maps layout for users to select their location and area of choice.
  2. Google Places API: This API will return the details of the places shown on Google Maps. The API provides all the necessary data for our app, including but not limited to: Name, Type, Coordinates, Address, Phone, Opening Hours, Rating, Price Level and etc.
