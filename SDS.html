1. Introduction

1.1 Purpose

The software design specification is made with the purpose of outlining the software architecture and design of the "Shuffler" in detail. The document will provide developers an insight in meeting client's needs efficiently and effectively. Morever the document facilitates communication and understanding of the software by providing several views of the system design.

1.2 Scope

The software design document would demonstrate how the design will accomplish the functional and non-functional requirements captured in the Software Requirement Specification(SRS). The document will provide a framework to the programmers through describing the high level components and architecture,sub systems,interfaces,database design and algorithm design.This is achieved through the use of architectural patterns,design patterns,class diagrams,database design etc.

1.3 Intended Audience

This document is mainly for the developers so that they can get a detailed idea about how the software need to be developed.

1.4 Definitions, acronyms, and abbreviations

SRS – Software Requirements Specification, this document which outlines the requirements that the application must fulfill. Entirely design independent.

User – any person who uses the program.

History – a database that will save the user activity history as strings.

Home Page – the initial Home page serves user to register/login.

User Menu Page – the initial menu for users who have logged in.

SDS – Software Design Specification.

Registration Page – the page that allows the user to create new acounts.

Login Page – the page that allows the user to login if they have a valid login.

UML – Unified Modeling Language.

OS - Operating System.

Activites - A list of cards that will display activities in a specified area.

1.5 Document Overview

The next chapter of the document has described architectural design of "Shuffler". The high level component and their interactions,suitable architecture patterns,physical arrangement of components and design decisions applied in the whole system.And the chapters after that described component and detailed design,class diagrams,class details,database design and many more.

2. Architectural Design

2.1 High Level Component and Their Interaction

User Component:

This is one of the key component of "Shuffler". This is where the user's actions, area activities will be implemented. This is also includes user account informations.

Authentication and user management component:

This is the major sub system that is responsible for the security of the "Shuffler" It authenticate users and also handles the user management activities such as creating new user account,removing acount etc.

History Component:

Here the user's recent history will be implemented. One user will have one history. It will be synchronized with server when a user logged in with a new device and it will be saved in local server if a user logs in with the previous device it will sync data from local server.

Activity Component:

In the activity component all activities of a speicified area will be saved into the database. User will be able to select a specific location and see the activities within that location.

2.2 Architectural Styles/Patterns

The Suffler will be developed under MVC(Model-View-Controller) architectural styles/patterns.Users will be able to access the application with active internet connection within any area of the world.

MVC Architectural Style(Model-View-Controller)

MVC Style separates presentation and interaction from the system data. The system is structured into three logical components that interact with each other.

 The Model component -Manages the system data and associated operations on that data.
 The View component- Defines and manages how the data is presented to the user.
 The Controller component- Manages user interaction and passes these interactions to the View and the Model.


2.3 Physical Arrangements of Device and Communication

The Shuffler deals mainly with hardware devices and installed software components on devices. The System performs many tasks. It consists of navigation based system used by users. The system helps to record user’s name, date of birth, email,phone number,address etc.
Internet is the worldwide interconnection of all smart communication devices that have a valid IP. There should be active internet connection to use the application.

2.4 Design Decisions Apllied in Whole Application

Object Oriented Software Development Methods

Reasons:

 Improved software maintainability.
 Faster development
 Lower cost development
 Improved software development productivity
 Higher quality software

MVC Architectural Pattern

Reasons:

 It should interact with other machines or users effectively.
 For more efficient interaction system should have flexible interfaces.
 MVC can be taken as for a popular and easy to handle web application development style that has the feature of separating the presentation, Business & intermediate logics.
 Ease to coding and provide well defined interfaces within each logic.
