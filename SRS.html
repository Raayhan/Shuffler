1. Introduction

1.1 Purpose

The purpose of this document is to build an android application called "Shuffler" that will help people acquire information about activities around his/her location. It will be available on Google Playstore. The build version of this software is version 1.0.

1.2 Document Conventions

The following SRS is organized independently by its index. Readers can easily access any of the specific information by following or clicking any of the index contents.

1.3 Intended Audience and Reading Suggestions

This document is intended for developers, testers, users, project managers and also for marketing stuffs. Readers will get a clear idea about the application fetures, external interface, design and implementation data, security details and many other contents.

1.4 Product Scope

The main purpose of the "Shufller" application is to provide users, information about activities of any area around their location. Users will be able to select any specific area or it will be automatically selected by GPS and after that they will see different activities thorugh cards and they will be able to select or shuffle any of those cards. It will be useful for all types of users. For example: travellers, students, teachers, doctors, drivers, businessmen etc.

2. Overall Description

2.1 Product Perspective

The product is entirely innovative, primarily targeting people who get stuck making choices when it comes to leisurely activities, and providing them an option to allow a 'randomizer' pick what activity they are to do. The activities are selected from the area of choice, and might therefore range from going to a restaurant to going skydiving.

The application will also save every user's choices, making it able to provide either a recommendation based on their own activities, or popular recommendations made on the area of choice.

Every activity will be shown as a card, full of necessary details such as time, cost and distance to travel. After a user chooses an activity, the counter shown on the card will go up, and the user will be prompted if they want directions towards the activity.

Saving user history will require a database. The GPS of the smart phone will also be required should a user ask for directions.





2.2 Product Functions

The user should be able to:

  • Select area to find activities in.
  • Shuffle or scroll through a list of activities.
  • Filter those activities by various criteria.
  • Shuffle or scroll through a list of locations for those activities.
  • Select a location of an activity.
  • Ask for directions to the location.
  • Check history.
  • Get recommendations based on self history.
  • Get popular recommendations based on area.
  • Report errors or missing funcionalities and activities.

2.3 User Classes and Characteristics

Interactors of the system are as follows:

  • Users of the application: Users will be able to use the app to find activities in a certain area. The acitivites can be filtered, and also be coming out of two types of recommender systems. The users can track themselves using their activity history, and also be able to ask for improvements to the app.
  • Admins of the system: Admins of the system will have direct access to the database of the system. They can create, remove or suspend accounts if they so choose. They will not be using the app, but rather a web portal provided for the database.

2.4 Operating Environment

The software will operate with the following software components and applications:

  • Application Platform: Android.
  • Hardware: Android smart phones.
  • Database: SQLite.
  • Administration Platform: Windows.
  • Coexisting Software: Google Maps.

2.5 Design and Implementation Constraints

  1. Internet connection is a necessity for the application to run, since it will be using Google Maps to fetch activity data.
  2. Availability of Google Maps to be used legally for the application is required.
  3. Database is required from both user and admin ends, and thus needs to give concurrent and fast access to data, especially since the activity tracking will get more extensive as more users engage with the app.
  4. GPS built in smart phones must provide decent accuracy for distance tracking after an activity is selected.

2.6 User Documentation

A tutorial for the user will be inside the app, once at the start, and later available in the options menu for revisiting. The tutorial will start off and go through the entire process of using the app.

2.7 Assumptions and Dependencies

  1. The first assumption is that the application will have to be distributed through the Google Play Store, as it will need to be on Android systems.
  2. The second assumption is the fair use of Google Maps as a service for the application itself, as well as the avaliability of certain details for each place in Google Maps in order to transform them into activities for the app.
  3. Database is required from both user and admin ends, and thus needs to give concurrent and fast access to data, especially since the activity tracking will get more extensive as more users engage with the app.
  4. GPS built in smart phones must provide decent accuracy for distance tracking after an activity is selected.

3. External Interface Requirements

3.1 User Interfaces

A sample user interface is provided as follows:





The user will be directed to the location page after starting the app, where they can select an area or confirm the automatically detected area. They will then be forwarded to the activity page, where a list of activities will show up as cards. This will be similar to the cards page as shown in the sample interface, when they can scroll or shuffle through the list of activities. Once they selecte an activity, they will be directed to the cards page, which will provide details of places that offer those activities. They can then choose to select a card, with the selection being saved, and then prompted for getting directions to the place.

3.2 Hardware Interfaces

The application does not have any direct hardware interfaces due to the lack of having any designated hardware other than android smart phones, which come built in with the capabilities to supply GPS data.

3.3 Software Interfaces

The application will be on Android, and will directly contact the SQLite database used to store user data, which consists primarily of login information and activity history of each user. The application will also be directly communicating with Google Maps for seleting location and detecting all places around that location. The implementation will be limited to how much data Google Maps provides on any place in the map.

3.4 Communications Interfaces

The application will use electronic forms to collect user login information for storage. Other than that, the application will only consist of selecting a location on the interface and scrolling or shuffling through the choices.

4. System Features

4.1 User Registration

4.1.1 Description and Priority

After the installation and launch of the application, the user will be presented with a welcome screen and would be prompted to create an account on the application server. The user will have to use the email associated with their Google account to get registered. After signing up, the user will be prompted to entr a unique User Name, which would be used as the handle for that particular user. After completing the process, an account for the user will be created to be stored on the server database and be used when necessary.

4.1.2 Stimulus/Response Sequences

Action

Response

1. Launch the Shuffler application from an Android device

1. Application is launched and shows the Welcome Screen, where the user is prompted to enter their email address and a password

2. Enter email and password

2. Information is sent to the server, and an account is created and stored in the database; The screen prompts the user to enter an User Name

3. Enter User Name

3. Information is stored in the database, and the user is taken to the main screen

4.1.3 Functional Requirements

REQ-1: The user must use a valid email address for the registration.

REQ-2: After registration the user's information must be securely stored in the serever database.

4.2 Location Based Activities

4.2.1 Description and Priority

The Shuffler application will be able to suggest activities based on the user's current or given location. The application will use an overlay over Google Maps for location selection, and will also need data maps for activity tracking. Based on the location, the user will get suggested activities available within the selected radius.

4.2.2 Stimulus/Response Sequences

Action

Response

1. Go to Location tab

1. The user will be prompted to enter their current or any other particular location.

2. Enter Location

2. The area would be calculated based on the given location and a 'Shuffle' menu will apppear, and the user will be shown the available activities within the selected area. The activities will apear in card form, displaying relevant details.

3. Choose to 'Shuffle'

3. The activities will be shuffled and will be displayed in a random order.

4. Select an Activity

4. The selected Activity Card will go up, and a message will be displayed: "Thank you for choosing this", as well as an option to get directions to the place of the selected activity.

5. Choose to get directions

5. A map will be displayed with highlighted directions to go to the selected place.

4.2.3 Functional Requirements

REQ-1: The Android device must have an active Internet connection

REQ-2: The Android device must have updated GPS services

REQ-3: The application must be synchronized to the user's Google account

5. Other Nonfunctional Requirements

5.1 Performance Requirements

  1. The application is dependant on internet connectivity, due to its direct connection with Google Maps. Therefore, a stable internet connection is required at all times.
  2. The GPS in the device for the application needs to be responsive to obtain the user's location automatically.
  3. There must also be a stable connection to the database to receive details of all the activities in the location selected by the user.
  4. The GPS built in smart phones must also provide decent accuracy for distance tracking after an activity is selected.

5.2 Safety Requirements

There are no negative effects on the user due to the use of the app. The applicaiton only tracks their behaviour while selecting activities. This data, even if lost, can be regained through further use of the app. The data leaking does not provide a massive threat either, since 'randomization', i.e. shuffling, is the key feature of the app.

5.3 Security Requirements

Due to the nature of the product, data of the user must be stored in order to provide recommendations. Therefore, to protect the data, multiple user authentication services are to be used, such as:

  1. Email or text verification while creating a user account.
  2. Email or text verification while resetting password.
  3. Email or text verification while deactivating account.
Since the product is to be made on the Android platform, any and all policies applicable to the platform will also be applicable to the product.

5.4 Software Quality Attributes

  1. Adaptability: The product adaps to all changes made on Google Maps, such as location/activity addition or removal.
  2. Availability: The availability of the product at all times is necessary for user satisfaction.
  3. Correctness/reliability: The product needs to be accurate about all the details it provides on the activities available, which in turn asks for its reliability.
  4. Interoperability: The application needs to efficient in exchanging information between itself and Google Maps, as well as itself and the underlying database.
  5. Portability: The product is on Android devices, which are primarily portable devices.
  6. Reusability: The user is to be drawn back to the product repeatedly for their needs.
